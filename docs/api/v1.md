# Stabletown API — v1 (CityChain Mirror Extract)

This mirror copy includes global conventions and CityChain challenge endpoints only.

# Stabletown API — v1

All endpoints are tenant-scoped and authenticated.

## Global conventions

### Contract source of truth

OpenAPI artifact:

* `contracts/openapi/openapi.v1.json` (generated)

Generated type artifacts:

* `frontend/operator-console/src/types/generated-api.ts`
* `sdk/packages/sdk/src/generated/openapi-types.ts`

Verification commands:

* `mix stabletown.openapi.verify`
* `mix stabletown.types.verify`

### Base URL

* `/api/v1`

### Auth

* `Authorization: Bearer <jwt>`

### Tenant scoping

One of:

* `X-Tenant-Slug: <tenant_slug>` header, or
* `X-Tenant-Id: <tenant_id>` header (current scaffold implementation path), or
* subdomain routing (e.g., `https://<tenant>.api...`)

**Rule:** every request must resolve `tenant_id` and all queries must filter by it.

### Idempotency (required for money-adjacent writes)

Money-adjacent endpoints must accept:

* `Idempotency-Key: <string>` header (recommended)

**Idempotent replay rule:** if a request with the same idempotency key is repeated, return the same semantic outcome without double-posting to the ledger.

### Time

* Timestamps are ISO8601 UTC.

### Money amounts

* Integers in cents: `amount_cents`
* Currency code string: `currency`

### Standard response envelope (recommended)

For consistency, prefer:

```json
{
  "status": "ok",
  "data": {}
}
```

### Errors

Use stable error codes:

* `unauthorized`
* `forbidden`
* `not_found`
* `validation_error`
* `insufficient_balance`
* `idempotency_conflict`
* `internal_error`

Example:

```json
{
  "status": "error",
  "error": {
    "code": "validation_error",
    "message": "name is required",
    "details": {"field": "name"}
  }
}
```

### POST /chains

Create a chain profile in the integration registry (`integration_type=chain`).

Additive chain profile runtime fields are supported:

* `runtime_kind` (`hedera | internal_mock | avalanche | evm | aztec`)
* `capabilities` (object)

```json

    "challenge_profile": "avalanche_build_games",

    "allowlist_policy": "managed",
    "validators": [
      {"name": "City Node 1", "role": "consensus", "status": "active"}
    ],
    "dependency_integration_ids": ["uuid"]
  }
}
```

Update chain profile metadata and dependency wiring.

`runtime_kind` and `capabilities` may be updated additively; when omitted, existing runtime metadata is preserved.

### POST /chains/:id/bootstrap

Execute CityChain bootstrap for a tenant-scoped chain profile and persist summary output.

**Required headers**

* `X-Tenant-Id`
* `Idempotency-Key`

**Request**

```json
{
  "bootstrap": {
    "network": "local",
    "strict": false
  }
}
```

**Response**

```json
{
  "status": "ok",
  "data": {
    "chain_bootstrap_run_id": "uuid",
    "chain_id": "uuid",
    "status": "completed",
    "network": "local",
    "rpc_url": "http://127.0.0.1:9650/ext/bc/C/rpc",
    "evm_chain_id": 43112,
    "contracts": {
      "institution_registry": "0x...",
      "merchant_registry": "0x...",
      "quest_manager": "0x...",
      "attestation_verifier": "0x...",
      "rewards_token": "0x...",
      "redemption": "0x..."
    },
    "seeded_roles": {
      "admin": "0x..."
    },
    "proof_mode": "deterministic",
    "proof_validated": false,
    "competition_grade": false,
    "deploy_artifact_path": null,
    "tx_receipts_artifact_path": null,
    "explorer_base_url": null,
    "deploy_tx_hashes": {},
    "flow_tx_hashes": {},
    "deploy_receipts": {},
    "flow_receipts": {},
    "explorer_links": {},
    "manifest_path": "/abs/path/chainkit/out/citychain-bootstrap-manifest.json",
    "summary": {},
    "metadata": {},
    "idempotency": "new"
  }
}
```

**Idempotency**

* Same key + same effective payload -> replayed bootstrap summary (`idempotency = replayed`)
* Same key + different payload -> `409 idempotency_conflict`

**Strict winner-grade mode (`strict=true`, network `fuji`)**

Strict mode hard-fails unless the bootstrap summary contains real on-chain proof:

* `proof_mode = "fuji_onchain"`
* `proof_validated = true`
* `competition_grade = true`
* `deploy_artifact_path` and `tx_receipts_artifact_path` (existing files)
* `deploy_tx_hashes` with deploy tx hashes by contract
* `flow_tx_hashes` for `quest_claim` and `redemption_chain_settlement`
* `explorer_links` for contract and tx verification

  * `contract_page`

  * `contract_page`
  * `quest_claim_tx`
  * `redemption_chain_settlement_tx`

**Idempotency**

* Same key + same payload -> replayed record (`idempotency = replayed`)

### POST /chains/:id/tx-intents

Submit chain transaction intent (idempotent).

**Required headers**

* `X-Tenant-Id`
* `Idempotency-Key`

**Request**

```json
{
  "tx_intent": {
    "operation_type": "quest_reward_transfer",
    "request": {
      "from": "ops_treasury",
      "to": "0xabc",
      "amount": "5000"
    },
    "amount_cents": 5000,
    "currency": "USDC",
    "metadata": {
      "provider_reference": "mock-runtime-1"
    }
  }
}
```

**AWS KMS Hedera operation (`kms_topic_create`)**

```json
{
  "tx_intent": {
    "operation_type": "kms_topic_create",
    "request": {
      "topic_memo": "stabletown-kms-bounty"
    },
    "metadata": {
      "custody_signer_id": "uuid",
      "hedera_account_id": "0.0.123456",
      "hedera_public_key": "302a300506032b6570032100...",
      "tenant_id": "uuid",
      "signing_algorithm": "ED25519_SHA_512",
      "message_type": "RAW"
    }
  }
}
```

Required metadata for `kms_topic_create`:

* `custody_signer_id`
* `hedera_account_id`
* `hedera_public_key`

Optional fields for `kms_topic_create`:

* `request.topic_memo`
* `metadata.signing_algorithm` (default `ED25519_SHA_512`)
* `metadata.message_type` (default `RAW`)

When the tx-intent finalizes, receipt metadata includes custody/KMS correlation fields:
`sign_job_id`, `aws_request_id`, `kms_key_id`, `signing_algorithm`, `hedera_tx_id`, `receipt_status`.

**Behavior (async finality lifecycle)**

* Default (sync) submit returns `status: "submitted"` and emits `chain.tx_submitted`.
* Feature-flagged async dispatch mode may return `status: "initiated"` with `dispatch_status: "queued"` and emits `chain.tx_queued`.
* Intent lifecycle remains within existing status vocabulary:
  * sync path: `submitted -> confirmed|failed`
  * async path: `initiated -> submitted -> confirmed|failed` (or `failed` on terminal dispatch failure)
* Finality outcomes are emitted asynchronously by the finality worker:
  * `chain.tx_confirmed`
  * `chain.tx_failed`
* Idempotent replay/conflict semantics remain unchanged.

**Aztec runtime scaffold notes**

* `runtime_kind=aztec` resolves to dedicated Aztec runtime module in runtime registry v2.
* Aztec runtime activation is tenant-flagged (`aztec_runtime_enabled`).
* Money-adjacent Aztec tx-intents can require compliance preflight (`aztec_compliance_preflight_enabled`):
  * missing verdict -> `409 validation_error`
  * expired verdict -> `409 validation_error`

**Additive observability fields**

The tx-intent response may include additive fields:

* `queued_at`
* `dispatch_status` (`queued|submitted|failed`)
* `dispatch_failure_reason`
* `dispatch_attempt_count`
* `dispatch_last_attempt_at`
* `dispatch_next_retry_at`

**Response (submitted)**

```json
{
  "status": "ok",
  "data": {
    "tx_intent_id": "uuid",
    "chain_id": "uuid",
    "status": "submitted",
    "failure_reason": null,
    "queued_at": "2026-03-02T00:00:00Z",
    "dispatch_status": "submitted",
    "dispatch_failure_reason": null,
    "dispatch_attempt_count": 1,
    "dispatch_last_attempt_at": "2026-03-02T00:00:00Z",
    "dispatch_next_retry_at": null,
    "submission": {
      "submission_id": "uuid",
      "tx_hash": "0xabc123...",
      "status": "submitted",
      "attempt_count": 0,
      "last_attempt_at": null,
      "next_retry_at": null,
      "last_error": null
    },
    "receipt": null
  }
}
```

### GET /chains/tx-intents/:id

Fetch tenant-scoped chain transaction intent detail, including submission and receipt snapshots.

**Behavior**

* While pending finality, `status` is `submitted` and `receipt` is `null`.
* After finality:
  * `status: "confirmed"` with receipt snapshot, or
  * `status: "failed"` with receipt snapshot and normalized `failure_reason`.
* Current normalized failure reasons:
  * `rpc_unavailable`
  * `submission_rejected`
  * `signature_failed`
  * `unsupported_operation`
  * `finality_timeout`
  * `unknown_runtime_error`

### POST /custody/sign-jobs

Submit signer job (idempotent).

**Required headers**

* `X-Tenant-Id`
* `Idempotency-Key`

When an active custody policy requires approval for the submitted `operation_type`, sign-jobs are persisted as:

* `status = requested`
* `approval_required = true`
* `approval_status = pending`

`sign_job.payload.message_base64` is optional. When present, custody providers sign those exact bytes (decoded from base64) instead of the synthetic payload blob.

**Request (`message_base64` signing path)**

```json
{
  "sign_job": {
    "signer_id": "uuid",
    "operation_type": "hedera_kms_topic_create_sign",
    "payload": {
      "message_base64": "aGVkZXJhLXR4LWJ5dGVz"
    },
    "metadata": {
      "chain_operation": "kms_topic_create",
      "hedera_account_id": "0.0.123456",
      "hedera_public_key": "302a300506032b6570032100..."
    }
  }
}
```

**Response excerpt**

```json
{
  "status": "ok",
  "data": {
    "sign_job_id": "uuid",
    "status": "signed",
    "signature": "base64-signature",
    "metadata": {
      "provider_metadata": {
        "aws_request_id": "request-id",
        "kms_key_id": "arn:aws:kms:...",
        "kms_key_arn": "arn:aws:kms:...",
        "signing_algorithm": "ED25519_SHA_512"
      }
    }
  }
}
```
